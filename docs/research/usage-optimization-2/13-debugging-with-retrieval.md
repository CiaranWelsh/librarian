# Debugging With Retrieval: Error-Message Search, Known-Issue Lookup, and Interleaving Search With Hypothesis Testing

**Sources (quoted figures attributed inline):** Agentless (Xia et al., arXiv:2407.01489 / FSE 2025); InferFix (Jin et al., arXiv:2303.07263); RAR for smart contracts (arXiv:2503.01098); RAGFix (NSF/PAR 10570230); ReDeFix (arXiv:2511.00678); WARP (arXiv:2509.25192); RelRepair (arXiv:2509.16701); "Reformulate, Retrieve, Localize" (arXiv:2512.07022); SWERANK+ (arXiv:2512.20482); ARISE (arXiv:2605.03117); SWE-Adept (arXiv:2603.01327); "How Many Tries Does It Take?" (arXiv:2604.10508); Debugging Decay Index (arXiv:2506.18403; *Sci. Reports* s41598-025-27846-5); FeedbackEval (arXiv:2504.06939); Self-Debugging / Self-Refine.

## 1. Debugging is a *different* retrieval regime from synthesis

Literature synthesis is shallow and one-shot (PaperQA2: ~1.26 searches/question). Debugging is the opposite: it is the one task where **interactive, multi-step retrieval reliably beats static one-shot RAG**. The "Reformulate, Retrieve, Localize" study reports that on bug localization, an interactive SWE-agent loop sits **27.0 / 28.0 / 21.0 points above BM25 RAG** on its three retrieval metrics — "the large benefit of the interactive agent loop over static retrieval." So the task-conditioned rule inverts: for debugging, *expect to search several times and let observations steer the next query*, not retrieve once and quote.

## 2. The error message is the highest-signal query — but raw, not rewritten verbatim

A recurring, effective pattern is **error-message-as-query**: feed the failing line / stack trace / compiler diagnostic straight into retrieval. RAR (smart contracts) uses "the executor's error messages... [the] faulty line is then used as a query to retrieve relevant code snippets." InferFix retrieves "semantically equivalent bugs and corresponding fixes" keyed off a static-analyzer error (e.g. null-deref at a line). RAGFix and ReDeFix retrieve community knowledge (Stack Overflow) against the error; ReDeFix lifts CSS-repair accuracy from **60.4% zero-shot to 88.3% with RAG**. This matches our Round-1 verbatim-beats-HyDE finding — the diagnostic *text* is already a dense, high-precision query — **but with one caveat**: unlike literature lookup, *query reformulation here helps*. "Reformulate, Retrieve, Localize" reports **up to +35% on first-file retrieval and up to +22%** end-to-end from rewriting the issue/error into a better localization query. The reconciliation: a clean compiler/stack-trace error is already a good verbatim query; a vague human bug report needs reformulation before it becomes one.

## 3. Two distinct retrieval targets: project code vs. known-issue knowledge

Debugging retrieval splits cleanly: (a) **project-internal localization** — find the faulty code (Agentless hierarchical localization; ARISE graph navigation; SWERANK+ multi-turn rank-and-aggregate); and (b) **external known-issue lookup** — find how this error has been fixed before (RAGFix/ReDeFix over Stack Overflow, WARP over web/docs, InferFix over a fix corpus). For the librarian, only (b) applies directly: it is a *known-issue / textbook-pattern* lookup — "what does the literature say about this failure mode / this algorithm's known pitfalls" — not code localization. The actionable implication is that the librarian's role in a debugging loop is the **InferFix/RAGFix slot**: retrieve a grounded explanation or canonical fix-pattern for an observed error, then let code-level tools (grep/LSP) handle localization.

## 4. Interleaving = ReAct + hypothesis/checkpoint; stopping is the hard part

The mechanism for "interleave search with hypothesis testing" is the ReAct loop: think → search → observe → revise. SWE-Adept makes the hypothesis explicit — after localization it "formulates one or more repair hypotheses," builds a per-hypothesis to-do list, and uses **checkpointing/branching to evaluate alternatives and revert failed edits.** SWERANK+ interleaves Search (retrieve candidate functions) with Aggregate (commit the best to persistent memory), progressively narrowing the root cause. The strong, quantified lesson is on **when to stop iterating**: self-repair gains decay sharply. "How Many Tries Does It Take?" finds **two repair rounds capture 76-95% of achievable gains** (and overall +4.9-17.1pp on HumanEval, +16-30pp on MBPP); FeedbackEval sees performance "stabilizing after two to three repair cycles"; the Debugging Decay Index reports models "lose 60-80% of their debugging capability within 2-3 attempts" (exponential decay). Self-Debugging capped 10 attempts but "successful debugging typically concluded within just three iterations." Logical/assertion errors resist repair (~45% fix rate) far more than syntax/name errors, and shallow feedback (stack traces without intermediate runtime state) causes "fixation & stagnation" loops. **Net rule: budget ~2-3 hypothesis-test cycles; a flat or worsening result is the signal to re-retrieve broadly (fresh start), not to patch again.**

## 5. Cost asymmetry justifies a tiered policy

Agentless shows a structured, *non-iterative* 3-stage pipeline resolves issues at **$0.34-$0.70/issue** vs agentic SWE-agent (~$1.05) / AutoCodeRover (~$1.33) — roughly **3-4x cheaper**, with localization itself proving "highly effective" even without agentic search (the bottleneck was patch *selection*, ceiling ~41%). The lesson for librarian usage: don't default debugging to an expensive open-ended loop. Start cheap (one targeted error-as-query lookup), escalate to multi-search only when confidence is low.

## 6. Actionable implications for task-conditioned librarian experiments

1. **Define a "debugging" task arm with error-as-query.** Test feeding a verbatim error/stack-trace as the librarian query vs. an LLM-reformulated query. Hypothesis (from §2): verbatim wins for clean compiler errors; reformulation wins (+20-35% analogue) for vague symptom descriptions. This refines, not contradicts, Round-1 verbatim-beats-HyDE.
2. **Cap hypothesis-test cycles at 2-3 and measure the decay curve.** Borrow the Debugging Decay Index: after each search-and-attempt cycle, log whether confidence/relevance improved. Test a "fresh-start re-retrieval" trigger (drop accumulated context, re-query broadly) against blind continuation — expect it to beat round-4+ patching.
3. **Make the confidence label the escalation gate.** Single error-as-query lookup first (cheap, the Agentless analogue); only if the top chunk's confidence is below threshold do we widen k or issue a second refined search. Benchmark this tiered policy on cost-per-resolved-question against an always-multi-search baseline.
4. **Interleave with breadcrumbs as the "citation-traversal" move.** When a retrieved chunk is high-confidence but incomplete, follow its breadcrumb to neighbouring chunks (the SWERANK+ Aggregate / PaperQA2 traversal analogue) rather than re-embedding a fresh query. Measure depth-vs-breadth: for debugging, expect *narrow-and-deep* (follow one promising thread) to beat *broad* better than it does for synthesis.
5. **Separate error-class outcomes.** Track librarian-assisted resolution by error type — conceptual/algorithmic ("known pitfall") vs. surface/API errors. §4 predicts the librarian helps most where the failure is a *known conceptual pitfall* documented in a textbook, least where it is an idiosyncratic local bug; report these separately so the abstention contract isn't penalised for refusing genuinely un-retrievable cases.
6. **Score the loop on cost-per-resolution, not retrieval recall.** Debugging's win condition is a correct fix, and the Agentless cost asymmetry shows cheap-first pays. Make cost-per-resolved-question (searches × tokens) a first-class metric for the debugging arm, distinct from the precision/recall metrics used for the synthesis arm.
