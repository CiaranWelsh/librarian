# Grounding Scientific Reasoning in Literature: Task-Conditioned Retrieval Usage

**Round-2 question:** Round 1 settled single-query mechanics (verbatim query, k=20 best / k=8 value, quote-first generation, abstention contract killing hallucination 12%->0%). This note asks how an assistant should *use* the librarian differently per task type, and what usage strategies (search count, refinement trajectory, breadth-vs-depth, stopping) are optimal.

## 1. Retrieve-or-not is itself task-conditioned — the strongest finding

The most decisive result for our purposes is that retrieval *helps factual/knowledge-intensive work and hurts pure reasoning*. On entity-centric QA, BM25+LLM lifts long-tail accuracy from ~15% to >45-50%, but **hurts frequent "head" queries when evidence is noisy** (the "override" effect) ([Retrieval-Augmentation survey, EmergentMind](https://www.emergentmind.com/topics/retrieval-augmentation)). Conversely, on math (GSM8K, MATH-500), static retrieval is *inferior to plain Chain-of-Thought*, and even adaptive traces that *do retrieve* underperform CoT while traces that *decline to retrieve* beat it — "retrieval only rarely helps reasoning... actively not using retrieval is indicative of good model performance" ([Adaptive Retrieval helps Reasoning—but mostly if it's not used, arXiv:2602.07213](https://arxiv.org/pdf/2602.07213)). Retrieval over raw docs is "corpus- and task-dependent and inconsistent"; OpenWebMath/StackExchange often *hurt* AIME while Wikipedia/ArXiv/GitHub alternate gain/neutral/regression by model ([RAG over Thinking Traces, arXiv:2605.03344](https://arxiv.org/html/2605.03344v1)). Implication: the decision to retrieve is a **metacognitive signal**, and frequency should scale with difficulty/knowledge-gap, not be applied uniformly.

For us this maps cleanly: **science/literature-synthesis = retrieve aggressively; maths/derivation = retrieve only for a named theorem/definition then reason unaided; coding = retrieve API/threshold facts, not logic.** Self-RAG was explicitly trained to retrieve *only* for knowledge-intensive instructions and skip it otherwise ([Self-RAG](https://selfrag.github.io/)), validating a per-task gate.

## 2. Retrieval granularity: finer is better, but search-unit ≠ generation-unit

"Dense X Retrieval" (EMNLP 2024) showed the indexed *unit* (passage/sentence/proposition) materially changes both retrieval and downstream QA. **Proposition-level indexing wins**: for a fixed word budget the answer appears in fewer retrieved words; the biggest proposition-over-passage gain is in the 100-200 word range (~10 propositions / 5 sentences / 2 passages) ([arXiv:2312.06648](https://arxiv.org/abs/2312.06648)). A controlled follow-up reports proposition indexing up to **+6.8 Recall@20 over passage on NQ (DPR), +10.5 points for Contriever** (72.1%->82.6%) — the gain is *larger for unsupervised retrievers* ([Passage/Sentence/Proposition empirical comparison, ResearchGate](https://www.researchgate.net/publication/404729110)). The standard reconciliation for long-form work is **parent-child / hierarchical** indexing: search on 200-400-token child chunks for precision, *hand the LLM the 800-1200-token parent* for coherence; select the parent when all children hit ([Hierarchical RAG pattern](https://cloudarcworks.io/resources/hierarchical-rag-pattern/); [LongRefiner, arXiv:2505.10413](https://arxiv.org/html/2505.10413v1)). For academic corpora specifically, structure-fidelity (preserve native section hierarchy, don't flatten) beats flat chunking, which "weakens section boundaries... and misallocates tokens away from evidence-bearing parts" ([SF-RAG, arXiv:2602.13647](https://arxiv.org/pdf/2602.13647)). Our markdown-chunks-with-breadcrumbs already encode this structure — the lever is *using breadcrumbs to widen to the parent section* for synthesis, while keeping the tight chunk for fact-lookup.

## 3. Search count & stopping: route by complexity, stop forward-lookingly

Adaptive-RAG routes once up front by a complexity classifier: **no retrieval (simple) / single-step (moderate) / multi-step iterative (complex)** ([Adaptive-RAG, NAACL 2024](https://medium.com/@tuhinsharma121/understanding-adaptive-rag)). This is the cleanest task-conditioning primitive. For multi-hop/synthesis, iterative retrieve-reason loops (IRCoT, Search-o1) decompose and refine queries with intermediate findings — but **each extra loop adds latency, cost, and distracting evidence**, so stopping matters more than retrieving. Confidence-based stops (FLARE token-probability, DRAGIN entropy+attention) and trained probers exist, but Stop-RAG shows present-focused signals mislead: "even if all necessary evidence is already retrieved, many irrelevant docs yield a low score and trigger further retrieval, which only adds noise." It frames stopping as a finite-horizon MDP with **forward-looking value** and beats fixed-iteration and prompt-based stopping on multi-hop QA ([Stop-RAG, arXiv:2510.14337](https://arxiv.org/abs/2510.14337)). Takeaway: **adaptive stopping is the underexplored lever** — judging when gathered info suffices.

## 4. Citation faithfulness: quote-first wins but trades against answer accuracy

ALCE is the standard (fluency/correctness/citation-quality, human-correlated) ([arXiv:2305.14627](https://ar5iv.labs.arxiv.org/html/2305.14627)). Two cautions for our quote-first contract: (a) **correctness ≠ faithfulness** — up to 57% of citations are post-rationalized (model answers from memory, token-matches a doc), which is "difficult to spot yet fosters misguided trust" ([Correctness is not Faithfulness, arXiv:2412.18004](https://arxiv.org/pdf/2412.18004)). (b) Verbatim/constrained-decoding citation (ReClaim) cuts citation length to ~22% of ALCE and reduces hallucination, **but shows an *inverse* relationship between answer accuracy and faithfulness** — tighter grounding narrows answer sources ([ReClaim, arXiv:2407.01796](https://arxiv.org/html/2407.01796)). PaperQA's search/gather-evidence/answer agent loop is the science-domain reference design ([arXiv:2312.07559](https://arxiv.org/pdf/2312.07559)); SciFact's three-stage (abstract retrieval -> rationale-sentence selection -> stance) is the verification template, with CliVER hitting 79.0% retrieval / 67.4% sentence-selection / 63.2% label precision ([SciFact/CliVER, PMC10919922](https://pmc.ncbi.nlm.nih.gov/articles/PMC10919922/)).

## Actionable implications for librarian usage experiments

1. **A/B a per-task retrieve-gate.** Maths/derivation and pure-logic-coding arms get a "retrieve only for named definition/theorem/API, else reason unaided" instruction; science/synthesis arms retrieve by default. Measure whether *declining* to retrieve on reasoning tasks improves correctness (predicted by arXiv:2602.07213).
2. **Test breadcrumb-driven granularity switching.** Lookup tasks: return tight top-k chunks. Synthesis/literature tasks: expand each hit to its parent section via breadcrumbs (parent-child). Measure coverage/diversity vs. precision and token cost.
3. **Route search-count by complexity (Adaptive-RAG style).** Single verbatim query for lookup; iterative decomposed queries for multi-hop synthesis; compare against a fixed-k=20 baseline on answer F1 *and* latency/token budget.
4. **Add a forward-looking stop rule for iterative arms.** Stop when a new query returns no novel chunk (overlap with already-cited) rather than at fixed depth; instrument the noise-injection failure mode from Stop-RAG.
5. **Score faithfulness, not just correctness.** Verify each emitted quote is a verbatim subsequence of a returned chunk; flag post-rationalization. Expect the ReClaim accuracy-vs-faithfulness tradeoff and quantify it on our confidence-labelled output.
6. **Use a SciFact-style decomposed metric** (chunk-retrieval / rationale-quote-selection / claim-stance) for the particle-physics collection so verification quality is separable from generation quality.
