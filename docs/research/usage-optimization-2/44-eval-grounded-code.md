# Evaluating Documentation-Grounded Coding Answers: API Hallucination With vs Without Docs Retrieval

**Scope.** Round-1 settled single-query mechanics for the librarian (verbatim query, k=20, quote-first, abstention contract). This note gathers the published evidence on the *coding* task specifically: how often LLMs hallucinate APIs, how much documentation grounding helps, where it *hurts*, and what that implies for a task-conditioned usage strategy. The headline result for coding is that grounding is not uniformly good — its value is conditioned on how well-known the target API is, which is exactly the kind of task-conditioning round 2 is about.

## How often do coding answers hallucinate APIs?

Hallucination rates are large and highly variable by model and by API popularity. In a misuse study across StarCoder, Copilot and Qwen2.5-Coder, hallucination was the single most frequent misuse class, reaching up to **43.3%** of parameter misuses (Java/StarCoder), dropping below 30% for stronger models ([Identifying and Mitigating API Misuse, arXiv:2503.22821](https://arxiv.org/html/2503.22821)). A practical-code study attributes **20.41%** of code hallucinations to API knowledge conflicts ([LLM Hallucinations in Practical Code Generation, arXiv:2409.20550](https://arxiv.org/pdf/2409.20550)). API-oriented evaluation found **4.0–9.9%** of generated examples simply omit the specified API ([arXiv:2409.15228](https://arxiv.org/pdf/2409.15228)).

The dominant predictor is **API frequency in training data**. On CloudAPIBench (AWS/Azure APIs annotated by public-domain frequency), GPT-4o reaches **93.66%** valid invocations for *high-frequency* APIs but only **38.58%** for *low-frequency* ones ([On Mitigating Code LLM Hallucinations with API Documentation, arXiv:2407.09726](https://arxiv.org/html/2407.09726v1)). This is the coding analogue of the "rare entity" problem: parametric knowledge is reliable for popular APIs and collapses for obscure ones — and a private detector/scientific corpus is overwhelmingly the obscure case.

## How much does documentation grounding help — and where does it hurt?

Documentation grounding helps most exactly where the model is weakest. On CloudAPIBench, Documentation Augmented Generation (DAG) raised GPT-4o low-frequency validity from **38.58% → 47.94%** (+9.36 pts). But with a sub-optimal retriever, DAG *dropped* high-frequency validity from **93.66% → 54.63%** — a **39.02 pt** absolute regression caused by irrelevant retrieved docs crowding out correct parametric knowledge ([arXiv:2407.09726](https://arxiv.org/html/2407.09726v1)). This is the cleanest published demonstration that **always-retrieve is the wrong policy**: forced grounding poisons answers the model already knew.

The fix is **selective, confidence-triggered retrieval (DAG++)**: retrieve only when the model's API-invocation confidence (minimum sub-token probability over the API name, well-calibrated, threshold ~0.7–0.8) is low. This fired retrieval **50.56%** of the time on low-frequency APIs but only **3.41%** on high-frequency ones, recovering high-frequency performance (94.15%) while net improving **+8.20 pts** overall ([arXiv:2407.09726](https://arxiv.org/html/2407.09726v1)). Other grounded settings corroborate the upside: structured-output RAG cut hallucinated steps from **~21% → <7.5%** and tables to **<4.5%**, and let a 3B model match a 15.5B one ([arXiv:2404.08189](https://arxiv.org/pdf/2404.08189)); a documentation-aware code RAG reports up to **60%** hallucination reduction ([arXiv:2407.09726 / DocPrompting line](https://arxiv.org/html/2407.09726v1)); domain RAG let 20–30B models beat a commercial baseline on quantum-library code ([QiboAgent, arXiv:2603.15538](https://arxiv.org/pdf/2603.15538)). Dependency-aware / execution-validated approaches (MARIN) beat naive RAG by **67.52% (MiHN)** and **73.56% (MaHR)** by validating that generated APIs actually exist ([arXiv:2505.05057](https://arxiv.org/pdf/2505.05057)) — grounding plus a verification step, not grounding alone.

## Breadth vs depth and the unanswerable case

More chunks is not better. Across retrievers, raising k introduces irrelevant/redundant context that dilutes signal; IRCOT peaked at ~10 docs and fell below the 5-doc level by 30 docs, and >60% of queries surface at least one highly distracting passage in the top-10 ([arXiv:2505.15561](https://arxiv.org/html/2505.15561), [arXiv:2512.14313](https://arxiv.org/pdf/2512.14313)). The "lost in the middle" U-curve degrades buried-context accuracy by >30% ([Liu et al. 2024](https://www.morphllm.com/lost-in-the-middle-llm)). For coding this matters doubly because file reads and search results accumulate over multi-step tasks. Consensus mitigation: retrieve broad, then **rerank/filter to ~3–5 chunks** and place the best first/last. This reconciles with round-1's k=20: retrieve k=20 for recall, but the *generation context* should be a reranked small set, not all 20.

Finally, coding queries are often **unanswerable** from the corpus (developer not proficient with the library). RaCGEval frames answerability as a 3-way classification (answerable / partial / unanswerable); baselines hit only **46.7%** ([arXiv:2411.05547](https://arxiv.org/html/2411.05547)). This is precisely the librarian's abstention contract, and it is *harder* for code than for prose because plausible-but-wrong code is cheap to emit.

## Actionable implications for librarian usage experiments

1. **Make the coding condition retrieval-mandatory and confidence-gated.** Detector/Rust-API answers are low-frequency by construction (parametric collapse to ~38%). But test a DAG++-style gate: when Claude is already confident about a stdlib/std API, skipping retrieval should not regress. Measure whether forced grounding *hurts* well-known-API answers.
2. **Separate retrieval-k from context-k.** Keep round-1's k=20 for recall, but A/B a rerank-to-3–5 generation step against pasting all chunks; measure hallucination and exact-signature accuracy, not just relevance.
3. **Add a verification step, not just retrieval.** MARIN's gains come from checking the API exists. For Rust, a cheap analogue is "does the cited symbol appear verbatim in a retrieved chunk?" — fold this into the quote-first contract as a hard gate.
4. **Use an answerability label as a first-class output.** Adopt RaCGEval's 3-way (answerable/partial/unanswerable) so partial-grounding produces a partial+caveat answer rather than abstain-or-hallucinate.
5. **Build a frequency-stratified eval set.** Mirror CloudAPIBench: tag target APIs/facts by corpus frequency so the grounding-helps vs grounding-hurts crossover can be located for *this* corpus and threshold tuned.
6. **Hold latency/cost as a reported axis.** Selective retrieval exists because always-retrieve is wasteful; report searches-per-task alongside accuracy so the coding strategy is justified on cost, not just correctness.
