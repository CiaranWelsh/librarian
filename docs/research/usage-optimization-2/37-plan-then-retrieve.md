# Plan/Outline-First Retrieval: Decompose-Then-Search as a Task-Conditioned Strategy

## 1. The core mechanism: separating decomposition from answering is what pays

The plan-first family all exploit one finding: an LLM that *first* lays out the sub-questions/plan, *then* answers, outperforms one that interleaves both. Press et al. name this the **compositionality gap** — the fraction of cases where the model knows every sub-fact but fails to compose them. Self-Ask (decompose the question into explicit follow-ups, answer each, then synthesize) lifts Bamboogle from **46.4% (chain-of-thought) to 57.6%** (+11.2pp), and Self-Ask **+ search** to **60.0%** (+13.6pp over CoT). The paper's own explanation is the load-bearing one for us: "the advantage of self-ask over chain-of-thought is that it disentangles the decomposition of the full question from the actual answers." Least-to-Most makes the same case at the extreme — explicit decomposition takes SCAN's length split from **16.2% (CoT) to 99.7%** with 14 exemplars. The lesson is not "plan is nice"; it is that the *plan is a different cognitive act* than retrieval, and doing it first removes a specific failure mode (missing-step / mis-composition).

## 2. Plan-first is most valuable for compositional and synthesis tasks, near-useless for atomic ones

The benefit is task-conditioned, and the size of the win tracks **how many sub-facts must be composed**. Plan-and-Solve quantifies the mechanism on maths: PS+ cuts GSM8K *missing-step* errors from **12%→7%** and calculation errors **7%→5%**, reaching **76.7%** vs 8-shot CoT's 77.6% with zero exemplars. But note *which* errors it fixes — missing-step (a planning/coverage failure), not semantic misunderstanding (which stays flat). This directly predicts the librarian split: plan-first helps where the answer is an *assembly* of retrieved pieces (literature synthesis, 4D-STEM-style investigation, multi-hop "how does X interact with Y") and is wasted overhead where the answer is a *single* retrieved fact (API signature, a threshold, one definition). Round-1's verbatim-single-query win is the limiting case of the same curve: for atomic lookups the optimal plan has one node, so "planning" collapses to "search once."

## 3. Outline-first writing: the plan is the retrieval scaffold AND the quality predictor

STORM is the strongest evidence that for *long-form writing*, the outline should precede retrieval and writing. STORM does pre-writing (discover perspectives → simulate multi-perspective Q&A grounded by search → synthesize an outline) *before* drafting section-by-section. Versus an outline-driven RAG baseline, STORM's articles are **+25pp organized** and **+10pp broad in coverage**, and critically the paper reports a **correlation between pre-writing (outline) quality and final article quality** — measured via heading soft-recall and heading entity-recall against human articles. So for writing tasks the plan is not just a search controller; outline quality is a *leading indicator* of output quality, which makes it directly instrumentable. This generalizes the production deep-research pattern (TTD-DR, NVIDIA AI-Q): a Planner emits the report skeleton first, and each section's heading becomes the seed for a targeted search query.

## 4. The decisive caveat: static plans lose to adaptive re-planning on long-horizon, environment-coupled tasks

Plan-first is not monotonically good. The Deep Research survey and Plan-and-Act both flag that a **static** upfront plan is "vulnerable to unexpected variations" — it cannot react to what searches actually return. MobilityBench makes this concrete: across models, **ReAct's final pass rate generally beats Plan-and-Execute** because its think-act-observe loop adjusts to tool output, whereas pure pre-planning "shows a significant lack" (best Plan-and-Execute FPR 65.8% vs best ReAct 69.1%). The resolution adopted by every strong 2025 system is **plan + dynamic re-plan**: TTD-DR and Step-DeepResearch keep the initial outline as scaffold but run a Replan/Reflect agent that "summarizes acquired information and adjusts research direction, reducing redundant searches." The actionable rule: the *value of upfront planning rises with compositional depth but falls with environment coupling/uncertainty* — for the librarian (a stable, read-only corpus with no irreversible actions) coupling is low, so a fairly complete upfront plan is cheap and safe, but it should still be revised when a sub-query abstains or returns low-confidence chunks.

## 5. Decomposition is bounded: cap the sub-question count

Self-ask/least-to-most decompose freely, but agentic RAG systems converge on tight caps because marginal sub-queries add latency without coverage. IterDRAG caps iterative decomposition at **≤5**; PaperQA2 runs *adaptive* search (≈5 searches, ~14.5 papers/question on LitQA2), not unbounded. Sibling file 21 records EfficientRAG hitting one-shot 200-chunk recall at ~20 chunks via a *trajectory* of small targeted queries. So the strategy is "decompose to a small fixed budget, then stop" — consistent with file 20's marginal-gain stopping rule.

## 6. Actionable implications for task-conditioned librarian experiments

1. **A/B plan-first vs query-first, conditioned on task family.** Hypothesis (from §1–2): plan-first wins on synthesis/multi-hop/investigation by a margin that *grows with the number of sub-facts to compose*, and is net-negative (pure overhead) on atomic lookup/API/single-threshold queries. Measure answer correctness AND total searches; the predicted crossover is the single-node-plan case.
2. **Instrument the plan as a measurable artifact, à la STORM.** For synthesis/writing tasks, have the agent emit an explicit sub-question list / outline before any search, then score plan coverage against the gold answer's required facets (analogue of heading entity-recall). Test STORM's claim that plan quality predicts final-answer quality on the librarian — if it holds, plan-coverage becomes a cheap early quality gate.
3. **Map task → decomposition budget and validate the cap.** Lookup → 1 query (no plan); maths/derivation → least-to-most chain; science/synthesis → 3–5 sub-questions (IterDRAG ≤5); long-form writing → outline with one search per section heading. Sweep the cap and confirm a knee around 3–5 (file 19's budget curves).
4. **Test static plan vs plan+replan against the abstention contract.** Because Round-1's abstention contract emits a confidence label per query, use *abstain / low-confidence* as the re-plan trigger: keep the upfront plan but re-derive remaining sub-questions when a node returns no usable chunks. Compare to a frozen plan; §4 predicts replan wins mainly when early searches surface unexpected gaps.
5. **Quote-first composition over the plan.** Round-1 showed quote-first generation wins; combine with plan-first by binding each sub-question's answer to its retrieved quote *before* synthesis (Self-Ask's disentangling, §1). Measure whether per-node quoting further cuts the compositionality gap vs quoting only at final synthesis.
6. **Use self-ask-style sub-questions as the query generator, not just a reasoning trace.** Each sub-question becomes one verbatim librarian query (preserving Round-1's verbatim-query and k=20 mechanics per search). Measure distinct-source coverage and searches-to-answer vs a single fat query — the EfficientRAG/file-21 prediction is the decomposed trajectory wins coverage at lower total chunk budget.
