# Collection Routing: When to Query Several Corpora, How to Merge, and Domain Classifiers

**Scope.** The librarian serves two heterogeneous collections (`software`, `particle-physics`). This note synthesises the literature on (a) when a task should hit one vs. several collections, (b) how to merge cross-collection results when scores are incomparable, and (c) whether to use a domain classifier vs. brute-force query-all. Round-1 already fixed single-query mechanics (verbatim query, k=20, quote-first, abstention); this is the routing layer above that.

## Key findings

**1. Querying everything is usually wasteful, and sometimes actively harmful.** The federated-RAG consensus is that "naively querying all silos introduces latency and noise" and is "often unnecessary." RAGRoute (Guerraoui et al., EuroMLSys '25, arXiv 2502.19280) showed a lightweight 3-layer NN router (256→128→1 logit, BCE-with-logits) that selects relevant corpora per query cut **total queries by up to 77.5% and communication volume by up to 76.2%** on MIRAGE/MMLU while **maintaining retrieval recall and answer quality**. Routing overhead was negligible: **0.3 ms on A100, 0.8 ms on CPU**. A separate harm argument comes from UniversalRAG (KAIST/DeepAuto, arXiv 2504.20734): merging heterogeneous corpora into one embedding space creates a **modality/domain gap** where retrieval is biased toward same-type content, so they *deliberately keep corpora separate* with independent embedding spaces and route with `r = Router(q)`. For the librarian this is the direct analogue: software and particle-physics chunks live in distinct semantic regions, and a software-engineering query that also pulls physics chunks is the failure mode to design against.

**2. Routing accuracy is the bottleneck, not routing in principle.** UniversalRAG's *trained* router is near-perfect in-domain but degrades out-of-domain (confuses Clip↔Document, Paragraph↔Document); its *training-free LLM* router is more balanced across distribution shift. Adaptive-RAG (Jeong et al., NAACL 2024, arXiv 2403.14403) trained a T5-Large complexity classifier (A=no-retrieval, B=single-step, C=multi-step) and its confusion matrix is sobering: **47% of no-retrieval queries misrouted to single-step; single↔multi confused 23%/31%**. Notably "results did not vary much based on the size of the model." Takeaway: a cheap LLM-as-classifier (the librarian already has Claude in the loop) is competitive with a trained router and degrades more gracefully — but misrouting is real and needs a hedge.

**3. The recommended pattern is route-with-fallback, not hard routing.** Across sources the robust production design is: classify the query → route to the 1–2 obviously relevant collections → **but fall back to query-all (or query-RAG) when the classifier is uncertain**. NVIDIA's RAG blueprint caps multi-collection retrieval at **≤5 collections** (reranker context blows up as `top_k × n_collections`) and makes a **reranker mandatory** when merging across collections. For a 2-collection system the cap is irrelevant, but the principle holds: when a query plausibly spans both, query both and let a reranker sort it.

**4. Merging cross-collection results requires rank fusion, not raw scores.** Scores from different indexes/embedding spaces are "fundamentally incomparable." The default robust answer is **Reciprocal Rank Fusion** (`1/(rank+k)`, k≈60): it needs no normalization, is outlier-robust, and is the standard for "multiple vector indexes." Montague & Aslam (2001) found normalization choice matters more than the fusion algorithm; Bruch et al. (2022) found tuned convex combinations of normalized scores (TM2C2) can generalize better and is more sample-efficient than RRF, at the cost of needing tuning. **Weighted RRF** lets you encode prior confidence per collection. A cross-encoder reranker as a final stage on the fused pool is standard (TREC iKAT 2025: fusion-then-rerank lifted nDCG@10 0.4218→0.4425) — but validate it, since cross-encoders can catastrophically demote bridge evidence in multi-hop settings.

**5. Task type changes the routing question, not just the depth.** The literature maps cleanly onto the task-conditioned framing: *single-fact lookup* (definition, API, threshold) → single best collection, one verbatim query, stop on first confident hit. *Literature synthesis / cross-domain* (e.g. "detector data-processing software" — genuinely spans both librarian collections) → query both, fuse, rerank. *Multi-step reasoning* → iterative retrieval with an explicit stopping rule. On stopping, the field has moved from fixed-iteration to adaptive: prompting-based (IRCoT stops at answer), confidence-based (FLARE retrieves when token prob < threshold), and value-based (Stop-RAG, arXiv 2510.14337, models the loop as an MDP and learns a Q-value stop controller, beating fixed and LLM-prompted stopping). The unifying lesson: **decide retrieval breadth from a cheap up-front query classification, and decide depth/stopping from a confidence signal during the loop.**

## Actionable implications for librarian usage experiments

1. **Test route-vs-query-all head to head on a 2-collection labelled set.** Build ~60 queries hand-labelled `software`-only / `physics`-only / `both`. Measure recall@k and answer correctness under three policies: (a) always query both + RRF, (b) Claude-classifier routes to one, (c) classify-then-fallback-to-both-on-low-confidence. RAGRoute predicts ~75% fewer redundant searches at preserved recall — verify whether the saving holds with only two collections (gains may be smaller than the federated multi-corpus case).

2. **Measure the misrouting cost explicitly.** Build a confusion matrix for Claude's collection classifier (cf. Adaptive-RAG's). The load-bearing number is the false-single rate on `both` queries — that is where a hard router silently drops evidence. If it is high, mandate the fallback-to-both policy.

3. **Adopt RRF as the default cross-collection merge and A/B it against quote-first-only.** Since the librarian returns top-k + a confidence label per chunk but scores across collections are incomparable, RRF (k=60) is the zero-tuning baseline. Test whether weighted RRF (prior weight per collection given the query's classified domain) beats unweighted.

4. **Make the breadth decision cheap and the depth decision a stopping rule.** Operationalise as: one up-front classification call sets breadth (1 vs 2 collections); the existing abstention/confidence contract doubles as the stopping signal for whether a second refined search is warranted. This mirrors Adaptive-RAG (pre-execution breadth) + FLARE/Stop-RAG (in-loop depth).

5. **Hold cross-domain queries out as their own condition.** "Software for physics detectors" is exactly the query class where single-collection routing fails and merge quality matters most — it should be a first-class experimental cell, not averaged into single-domain results.

6. **Cap refinement, not just k.** Round-1 fixed k; this round should fix a per-task search budget (e.g. fact lookup = 1 search; synthesis = 2–3 + fuse; reasoning = adaptive stop) and measure marginal correctness per extra search to find the value point, mirroring Adaptive-RAG's "match multi-step accuracy at ~half the steps."
