# Grounding Design Decisions in Literature: Task-Conditioned Librarian Usage for Architecture/ADR Work

**Sources:** Skarlinski et al. arXiv:2409.13740 (PaperQA2, cross-ref file 10); Dhar et al. "Can LLMs Generate Architectural Design Decisions?" arXiv:2403.01709; "Context Matters: Context Strategies for Automated ADR Generation" arXiv:2604.03826; "Using LLMs in Generating Design Rationale for Software Architecture Decisions" arXiv:2504.20781; "Evaluating LLMs for Detecting Architectural Decision Violations" arXiv:2602.07609; "Understanding the Design Decisions of RAG Systems" arXiv:2411.19463; RAG robustness survey arXiv:2506.00054; Nygard (2011) / adr.github.io / Microsoft Well-Architected ADR guidance; AWS Architecture Blog (3-yr ADR practice). Corpus grounding texts (librarian, collection `software`): Bass, Clements & Kazman *SAIP* Ch.20 (ADD Step 4, choose design concepts) & Ch.21 (ATAM); Cervantes & Kazman *Designing Software Architectures* Ch.4, Ch.11.

## 1. Why "grounding a design decision" is a *different task class* from literature QA

Literature QA (file 10) is recall-of-fact: PaperQA2 needs ~1.26 searches because the answer lives in one place. A **design-decision task** is not a lookup — it is *selecting among alternatives under quality-attribute trade-offs and recording the rationale*. SAIP Ch.20 (ADD Step 4) and Cervantes & Kazman Ch.4 frame this as enumerating candidate design concepts (reference architectures, patterns, tactics, externally-developed components) and choosing against drivers; SAIP Ch.21 (ATAM) frames evaluation as walking a quality-attribute **utility tree** to find sensitivity and trade-off points. The literature is therefore consumed in *three distinct retrieval shapes*, and the optimal librarian strategy differs for each:

- **(a) Pattern/tactic lookup** — "what design concept solves problem X." Convergent, near-QA. One or two targeted searches; the catalog entry (GoF pattern, SAIP tactic) is a localized chunk.
- **(b) Trade-off analysis** — "what does X cost me on availability vs. modifiability." Divergent; the answer is *distributed across multiple sources* (one per quality attribute) and is inherently multi-perspective. Demands breadth-first, multi-source retrieval.
- **(c) Rationale/ADR authoring** — "write the decision + justification + consequences." Synthesis under a fixed template; the binding constraint is *project context*, not corpus breadth.

## 2. Concrete numbers, by task shape

**Pattern lookup → near-QA, shallow.** Design-pattern recognition/recommendation work (Empirical Software Eng. 2025; IET Software 2026) treats it as embed-and-kNN retrieval against a curated catalog (P-MARt). It behaves like file 10's lookup regime: broad-retrieve / narrow-answer, one search. The IET vector-DB study is essentially RAG with metadata-carrying chunks — directly analogous to our librarian and confirms a single semantic query suffices when the target is a discrete catalog item.

**Rationale/ADR authoring → context strategy dominates over corpus breadth.** "Context Matters" (arXiv:2604.03826) is the sharpest result for us. It compared five context strategies for ADR generation: Baseline (none), All_N (full history), First_K, Last_K, and RAFG (retrieval-augmented few-shot). Findings: providing *any* grounding context lifted BLEU ~4× (Gemini-2.5-Pro: 0.0193 → 0.0755 with All_N). But **a small recency window matched exhaustive context**: Last_K(3) BERTScore-F1 0.8440 (peak), and on Qwen3-235B Last_K(3) 0.8375 ≈ All_N 0.8379, while First_K(3) trailed at 0.8344. Grounding also *reduced verbosity*: baseline 1,123 tokens vs. ground-truth 527; grounded 637–776. Implication: for authoring, **a few highly-relevant exemplars beat dumping everything** — the same "narrow answer-set wins" lesson as PaperQA2's 5-summary optimum, and as our Round-1 k=8 value point.

**Generation feasibility → strong but sub-human, context-sensitive.** Dhar et al. (arXiv:2403.01709): zero-shot GPT-4 generates "relevant and accurate" decisions but below human level; GPT-3.5 matches it few-shot; Flan-T5 matches after fine-tuning. So *exemplar retrieval (few-shot) substitutes for model scale* — retrieval quality is the lever.

**Trade-off / rationale breadth → RAG help is real but hard to attribute.** The multi-agent DR study (arXiv:2504.20781) retrieved aggressively (60 URLs/aspect → FAISS-filtered to top-10) yet concluded "the impact of external knowledge is difficult to discern." Its verification agent drove "Misleading" arguments down to 1.59%; recall reached 0.715 but precision was only 0.267 (F1 0.389). Lesson: for divergent trade-off work, **more retrieval did not cleanly help; an explicit review/verification pass did** — echoing PaperQA2's rerank-summarize-verify spine.

**Detection/validation → high-precision, single-pass.** Decision-violation detection (arXiv:2602.07609; 980 ADRs, 109 repos, screen-then-3-validators) achieved strong accuracy on "explicit, code-inferable" decisions. This is a fourth task shape (audit an existing decision against the corpus) — high precision, low retrieval depth, multi-model cross-check rather than multi-search.

## 3. The trade-off-analysis strategy the SE literature actually prescribes

ATAM (SAIP Ch.21) and the analysis chapter of Cervantes & Kazman (Ch.11) say a trade-off question is answered by *scenarios across multiple quality attributes*, not one fact. The retrieval analogue: a single dense query under-serves it because each QA branch (performance, availability, modifiability, security) is a different region of the corpus. This predicts trade-off tasks need **decomposed, multi-query (one per quality attribute) breadth-first retrieval with source diversity as the stop signal** — the opposite of pattern lookup's single shot. The RAG design-decision survey (arXiv:2411.19463) and robustness survey (arXiv:2506.00054, naming "context compression / utility filtering" and "retrieval-conditioned generation control" as recurring patterns) corroborate that *control logic over retrieval*, not raw k, is where task-conditioning lives.

## 4. Actionable implications for librarian usage experiments

1. **Branch the strategy on task shape, not on query text.** Define four librarian "modes" — *lookup* (1 search, broad-retrieve/narrow-answer), *trade-off* (N parallel searches, one per quality attribute, source-diversity stop), *authoring* (retrieve few exemplars + project ADRs, Last_K-style recency window), *audit* (single retrieve + multi-pass self-check). Measure each against a fixed single-shot baseline.
2. **For ADR authoring, test "few exemplars" vs. "dump all."** Replicate "Context Matters" on the librarian: feed the 3 most-similar prior ADRs/decision chunks vs. all retrieved chunks. Hypothesis (from BERTScore parity above): 3 exemplars match full context at far lower tokens and reduce verbosity. Track output length against a target as a quality proxy.
3. **For trade-off questions, decompose by quality attribute before retrieving.** Issue one query per ATAM utility-tree branch rather than one fused query; measure coverage of distinct trade-off/sensitivity points recovered vs. single-query. Stop when ≥M distinct sources span the relevant QAs.
4. **Add a verification pass, not more searches, for divergent tasks.** The DR multi-agent and violation-detection results both show a review/cross-check step beats deeper retrieval. Test a librarian "answer → re-query to confirm each rationale clause is cited" loop, measuring unsupported-clause rate (precision) — the axis Round-1's abstention contract already protects.
5. **Make the confidence label task-sensitive.** For lookup, a high-confidence top chunk should hard-stop. For trade-off/authoring, a single high-confidence chunk is *insufficient* (single-source over-reliance is the failure mode); require diversity before allowing generation.
6. **Score by task-appropriate metric.** Lookup/audit → precision and exact-match; authoring → semantic similarity to gold ADR + verbosity; trade-off → recall of distinct trade-off points. Do not use one metric across modes — the studies above each chose metrics matched to their task, and a uniform metric would hide the task-conditioning effect we are trying to measure.
